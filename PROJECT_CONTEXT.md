## 1. Product Summary
IBMiHub AI is an AI-powered SaaS platform for IBM i professionals. The platform is intended to support learning, practice, AI assistance, documentation, debugging, modernization, and enterprise training.

---

## 2. Founder Role
The founder is the Product Owner and decision-maker for product direction, scope, and priorities.

---

## 3. AI Assistant Roles
The project uses multiple AI roles and responsibilities:
- Product Architect: responsible for product vision, business decisions, SDD specifications, and architecture direction
- Lead Software Engineer / Technical Architect: responsible for engineering quality, implementation readiness, and technical execution
- Engineering assistants: support documentation, planning, review, and execution within approved scope

---

## 4. Current Phase
Current phase: Sprint 0 Planning

The team is preparing the documentation, engineering foundation, and planning artifacts before implementation begins.

---

## 5. MVP Scope
The MVP is expected to focus on a narrow set of early capabilities, such as:
- user access and dashboard foundation
- learning center
- AI tutor
- progress tracking

The exact Sprint 1 scope is still being finalized.

---

## 6. Tech Stack Direction
The current technical direction includes:
- modern web application stack
- AI integration for IBM i-focused assistance
- scalable, maintainable architecture for long-term SaaS growth

No application code is being created in this phase.

---

## 7. Development Process
The project follows a Spec-Driven Development approach.

Each feature should be supported by:
- a specification document
- a plan
- a task list
- test cases

Documentation and architecture planning are prioritized before implementation.

---

## 8. Key Rules
- Do not change product requirements without approval.
- Do not invent new features.
- Do not remove existing features without approval.
- When requirements are unclear, ask questions instead of guessing.
- Suggest improvements, but do not implement them without approval.
- Treat SDD documents as the source of truth.
- Keep architecture and implementation separate.
- Prioritize readability, maintainability, and security.
- Respect the original-content/no-copying policy.
